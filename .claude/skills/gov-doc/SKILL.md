# 공공/정부과제 제출용 개발 기록 보고서 작성

## 언제 쓰나

사용자가 **공공기관·정부과제에 제출할 개발 기록 문서**를 요청할 때. 보통 "이 코드/모듈을 근거로 제출용 문서를 만들어줘", "공공문서 양식으로 작성해줘", 목차 항목(예: `4.2.1 …`)을 주며 "각각 문서로 만들어줘" 형태로 온다.

핵심 성격: **실제 코드를 근거로 하되, 읽는 사람은 개발자가 아니다.** 그래서 (a) 사실은 코드에서 가져오고, (b) 표현은 소스 파일명·내부 식별자를 감추고 의사코드·다이어그램·설계 논거로 재구성한다.

## 절대 규칙 (사용자 피드백으로 확정된 것 — 반드시 지킬 것)

1. **실제 소스 파일명을 본문에 노출하지 말 것.** `구현 위치: service.py` 같은 라인, 부록의 파일 목록(`config.py — …`) 금지. 검토자는 그 파일이 어디 있는지 모르므로 의미가 없다.
   - 대신 **기능 단위**로 서술한다: "탐지 사이클 오케스트레이션", "다중 카드 분산 추론기" 등.
2. **코드 스니펫은 허용되지만 반드시 의사코드(pseudocode)로.** 실제 소스를 그대로 붙이지 말고 핵심 로직만 한국어 의사코드(`함수`/`반환`/`만약`)로 요약한다. 문서 상단과 §5 도입부에 "실제 소스가 아니라 의사코드" 고지를 넣는다.
3. **내부·제품 고유 명칭은 일반화한다.** 예: 특정 모델 코드명(`PE-Core-L14-336`) → 성격 기반 명칭(`CLIP 기반 PIA 모델`). 단, 사용자가 문서 제목/요청에 특정 이름을 명시했으면(예: `Qwen3-VL`) 그건 그대로 쓴다. 애매하면 사용자에게 확인.
4. **민감정보는 플레이스홀더 `<...>`.** 카메라 주소·자격증명·내부 호스트·경로 등.
5. **다이어그램은 Mermaid + ASCII 병기.** 렌더 안 되는 환경(한글 뷰어/출력물) 대비.
6. **제목(H1)에는 목차 번호를 빼고, 파일명에는 번호를 넣는다.** 파일 `4.2.1_이벤트별_….md`, H1 `# 이벤트별 …`.
7. **커밋하지 말 것.** 사용자가 직접 편집·제출하는 문서다(명시 요청 없으면 git 건드리지 않음).
8. **작성 전 실제 코드를 읽어 사실 근거를 확보할 것.** 추측으로 기능·수치를 쓰지 않는다.

## 작성 절차

1. **대상 코드 읽기.** 사용자가 가리킨 모듈/패키지 경로의 핵심 파일을 읽어 실제 동작을 파악한다(진입점 → 설정 → 핵심 로직 순). 성능 수치·정확도 지표가 이미 리포트/주석에 있으면 그 값을 근거로 인용.
2. **범위 확정.** 목차 항목이 여러 개면 각 항목당 1개 문서. 각 문서가 다루는 경계(무엇이 이 문서 범위이고 무엇이 옆 문서인지)를 §2에서 명시.
3. **명칭 매핑 결정.** 내부 명칭 → 일반화 명칭 표를 머릿속에 잡고 문서 전체에서 일관 적용. 사용자가 이전에 준 매핑(예: PE→CLIP 기반 PIA)을 재사용.
4. **템플릿(`TEMPLATE.md`)의 뼈대**로 각 문서를 채운다. 섹션은 가감 가능하나 순서·톤은 유지.
5. **성능 수치 포함 여부는 상황 판단.** 측정 리포트가 있으면 근거 인용하되, 공공 제출에서 수치 인용이 부담되면 정성 서술로 바꿀지 사용자에게 한 번 확인하는 게 좋다.
6. **검증.** 작성 후 `grep -nE "\.py|구현 위치"` 로 파일명·구현위치 잔존을 확인(clean이어야 함). 일반화하기로 한 내부 명칭도 grep으로 잔존 확인.

## 문서 구조 (표준 순서)

상세 뼈대는 같은 폴더의 **`TEMPLATE.md`** 참조. 요약하면:

- **제목 + 부제** (부제 = "… 개발 결과 보고서")
- **요약표** (대상 시스템 / 개발 구분 / 대상 모델·가속기 / 문서 목적) — 필요 항목만
- **의사코드 고지** (blockquote)
- **1. 개요** — 1.1 배경 및 필요성 / 1.2 개발 목표 / 1.3 결과 요약
- **2. 시스템 내 위치** — 전체 파이프라인에서 이 모듈의 위치(Mermaid+ASCII), 이 문서의 범위 경계
- **3. 아키텍처 / 스키마 / 데이터 플로우** — 대상에 맞춰 제목 조정
- **4. 개발 내역** — 단계별(1→2→3단계) + **4.x 대표 성과**(번호 목록)
- **5. 핵심 구현 (의사코드)** — 기능별 의사코드 블록(파일명 없이)
- **6. 설계 결정 및 트레이드오프** — `대안 | 기각 사유` 표 + "→ **선택:** …" + **위험요인 및 완화** 표
- **7. 검증 결과** — 기능/정확도/성능(있으면 수치)
- **8. 결론 및 향후 계획**
- **부록 A. 구성 요소 및 자산** — 기능 단위 목록(파일명 금지) + 데이터/입력 계약 + 대상 이벤트
- **부록 B.** (선택) 구성(env)·정책 요약 표
- 맨 끝 이탤릭 재고지: *"본 문서의 코드 예시는 의사코드이며 …"*

## 톤·표현

- 공식 보고서 문체(평서·간결), "본 사업을 계기로 …", "… 를 신규 개발/정의하였다".
- 각 설계 결정은 **대안을 기각한 논거**로 정당화(공공 검토 관점).
- 여러 문서를 만들면 서로 `(문서 4.1.1)`처럼 교차 참조해 하나의 체계로 보이게 한다.
