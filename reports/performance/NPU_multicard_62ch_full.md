# [full NPU] PE-Core-L14-336 멀티카드(7×ARIES) 멀티채널 배치 지연시간 벤치마크

`NPU_multicard_62ch_benchmark.md`(당시 hybrid: NPU trunk만 측정, CPU pool 별도)의 **full NPU 판**.
**동일 테스트 구조**로, 이번엔 추론 단계 I가 **full NPU MXQ(trunk + attn_pool, QKᵀ16bit) = image→embedding
전부 NPU**다. 배치로 N채널이 한꺼번에 들어올 때 7대(56코어)에 분산, 1→62채널 1채널씩 측정.

| 단계 | 정의 | 자원 |
|---|---|---|
| **P 전처리** | 원본 이미지 N장 → 모델 입력(HWC float32). resize 336 + normalize | CPU |
| **I 순수추론** | 모델 input → **embedding** (full NPU: 24 block + attn_pool, QKᵀ16bit) | NPU (7대 분산) |

> **vs 비포**: 비포(hybrid)의 I는 trunk(24block)만이고 CPU pool head(attn_pool ~2ms/장)가 별도였다.
> full NPU는 **attn_pool이 I에 흡수**됐는데도 I 시간이 비포 trunk와 거의 같다(아래) → head NPU 추가비용 ≈0.
> 정확도: 원본 PyTorch 대비 **cos 0.99**(COCO calib). 원인·해결: [`../vendor/mobilint_resolution_attn_pool.md`](../vendor/mobilint_resolution_attn_pool.md).

## 2. 핵심 결과 (headline)

- **full NPU 추론도 7대로 거의 선형 확장** — 56채널을 **525ms**에 처리(채널당 9.4ms, **106.7 img/s**). 1대 대비 **×6.8**.
- **1~7채널은 ~287ms로 평탄** — 7대에 1장씩 꽂으면 단건(286ms)과 동일. 완벽 병렬.
- **추론 knee = 56채널(=56코어).** 57채널부터 일부 NPU가 9장(>8코어)을 받아 2번째 웨이브 → ~796ms로 계단 상승.
- **★ 비포(trunk-only) 대비 I가 거의 동일**: 56ch 525ms(full) ≈ 519ms(trunk). **attn_pool을 NPU에 올렸는데 추론시간 추가 ≈0** (head는 전체 연산의 ~0.8%). 즉 **CPU pool head를 없애고도(=full NPU) 추론 처리량 손실 없음.**
- **고채널 병목은 여전히 전처리(CPU)** — full NPU도 추론(525ms)은 빠르고, 4K→336 resize가 채널 수에 선형. 전처리 병렬화가 다음 1순위(`NPU_preprocess_parallel.md`).

## 5. 해석

### 순수 추론 (full NPU)
- **per-NPU 부하 = ⌈N/7⌉.** N≤7이면 1대당 1장 → 287ms(=단건). 8코어 async가 부하를 흡수.
- **56채널까지 1대당 ≤8장 → ~525ms.** 채널당 실효 추론 284ms(1ch) → **9.4ms(56ch)** 로 30배 개선.
- **57채널부터 계단 상승**(525→796ms): 1대당 9장(>8코어) → 9번째가 다음 웨이브. 운영 권장 상한 = **56채널/배치**.
- **★ full(trunk+attn_pool)인데 비포 trunk-only와 I가 동일**(56ch 525≈519). attn_pool을 NPU로 올린
  추가비용이 사실상 0 → **CPU pool head 제거(=full NPU)의 추론 비용은 무료.**

### 전처리 (CPU) — 여전한 병목
- 4K→336 resize가 채널 수에 선형. full NPU에서도 추론(525ms)보다 전처리가 크다 → 전체 배치시간 좌우.
- 개선(미적용): 멀티프로세스/스레드 resize, 입력 해상도/디코드 통합, 전처리-추론 파이프라이닝.

### 종합
- 7대 분산으로 **full NPU 추론 ≈ 107 img/s**, 56채널 동시 추론을 약 0.52초에. **CPU pool 없이** 달성.
- end-to-end는 전처리 병목 → 전처리 병렬화가 다음 최적화 1순위.
