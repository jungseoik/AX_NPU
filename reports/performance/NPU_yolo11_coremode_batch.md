# YOLO11 NPU — 컴파일 · 코어모드 배치지연 · 카드 스케일링 · 정확도(mAP)

PE-Core(비전인코더) 외에 **YOLO11 객체탐지**를 ARIES NPU로 직접 컴파일·추론한 실측.
코드: `yolo_npu/` (compile/detect), 튜토리얼: `tutorial/yolo_npu/`.
**4 사이즈(11n/11s/11m/11l) × 4 코어모드(single/multi/global4/global8) 전부** 컴파일해 측정.

## 1. 컴파일 — 패치 불필요

- 경로: ultralytics `yolo11*.pt` → ONNX(imgsz 640) → `mxq_compile(backend="onnx", yolo_decode_include=True)`.
- **PE와 달리 모델 패치 0개.** YOLO11은 표준 CNN 검출기라 qbcompiler가 바로 파싱,
  YOLO decode까지 그래프에 포함 → 출력 `(1,8400,84)` (anchor 8400 × [cxcywh+80 class]).
- INT8 MXQ, calib = COCO val2017 200장(PE와 동일 방침). 컴파일 시간(호스트 CPU, single):

| 모델 | MXQ 크기 | single 컴파일 |
|---|--:|--:|
| yolo11n | 6 MB | 133 s |
| yolo11s | 12 MB | 219 s |
| yolo11m | 23 MB | 229~279 s |
| yolo11l | 29 MB | 341 s |

- 4 코어모드(single/multi/global4/global8) 각각 `inference_scheme`로 생성.

| 모델 | 단건 저지연(global8) | 배치64 global4 | 배치64 처리량 | 채널당(수렴) |

| yolo11n | 6.7 ms | 119.5 ms | **536 img/s** | 1.9 ms/img |
| yolo11s | 5.1 ms | 156.7 ms | **408 img/s** | 2.4 ms/img |
| yolo11m | 9.0 ms | 262.6 ms | **244 img/s** | 4.1 ms/img |
| yolo11l | 11.2 ms | 327.1 ms | **196 img/s** | 5.1 ms/img |

- **모드 선택은 사이즈 무관하게 동일**: 처리량/다채널 = **global4**(단건도 빠르고 고배치 최속), 단건 최저지연 = **global8**, **multi는 전 구간 최악**(쓰지 말 것).
- **single**은 8코어를 멀티스레드로 채워야 global4에 근접(단일스레드면 코어 1개만 씀). 고배치에서 global4와 비슷하나 저배치가 느림.
- 카드당 동시 병렬은 코어 배분 한계로 배치가 커지면 선형 증가(코어 다 참). 더 필요하면 카드 추가(3절).
- 사이즈 간 비율 ≈ 11n:11s:11m:11l = **1 : 1.3 : 2.2 : 2.7** (채널당 기준). 참고로 PE(ViT-L) ~63ms/img → 11l조차 12배 이상 가볍다.

| 모델 | 1장 | 2장 | 4장 | 7장 | 7장 speedup | 7장 처리량 |

| yolo11s | 156.3 | 71.1 | 41.7 | 27.5 | **5.7x** | **2327 img/s** |
| yolo11m | 257.8 | 120.2 | 66.6 | 43.5 | **5.9x** | **1471 img/s** |
| yolo11l | 322.3 | 156.3 | 82.6 | 51.5 | **6.3x** | **1243 img/s** |

- **고배치일수록 카드 수에 거의 선형**(7장 speedup 5.3~6.3x). 무거운 모델일수록 선형에 근접(작은 모델은 배치가 작을 때 스레드/런타임 오버헤드가 비율상 커서 손해).
- **저배치(1~4)는 카드를 늘려도 이득 미미**(단건은 어차피 코어 1~4개면 충분) → 카드 추가는 **고배치·다채널일 때만** 효과.
- 채널 수 ≤ 카드 수면 카드당 1장씩 완전 병렬 → 배치 1~7 구간이 거의 평평(단건 지연 유지).

## 4. 멀티카드 출력 무결성 (검증)

`../scripts/verify_yolo_multicard_output.py`: 서로 다른 val2017 **40장**을 단일카드 기준 vs
2장·7장 분산 `detect_batch`로 위치별 비교(개수+클래스+좌표+conf) → **0/40 불일치**(완전 일치).
순서 보존 + 카드 간 오염 없음. **async(multi-in-flight)는 출력이 깨져** 안 쓰고 멀티스레드 동기 infer를 쓰는 이유(PE와 동일 방침, `NPU_pe_throughput_modes_full.md`).

- **e2e(전처리+NMS 포함)는 2장 이후 CPU 전처리 병목**(순수 NPU는 카드에 비례하지만 e2e는 전처리에서 평평해짐) — 고배치 다음 최적화는 전처리 병렬화(`NPU_preprocess_1_parallel.md`).
- 사용: `YOLONPU(mxq, device_ids=[0,1])`(지정) / `device_ids="auto"`(전체) + `detect_batch([...])`.

## 5. 정확도 (mAP, COCO val2017 300장)

우리 NPU MXQ(INT8) vs fp32(onnxruntime), **동일 이미지·동일 전/후처리**, pycocotools.

| 모델 | | mAP@0.5:0.95 | mAP@0.5 |
|---|---|--:|--:|
| **yolo11n** | fp32 baseline | 0.4335 | 0.5827 |
| | **NPU INT8** | **0.4215** | **0.5728** |
| | 양자화 손실 | **−2.8%** | −1.7% |
| **yolo11s** | fp32 baseline | 0.5084 | 0.6750 |
| | **NPU INT8** | **0.4932** | **0.6602** |
| | 양자화 손실 | **−3.0%** | −2.2% |
| **yolo11m** | fp32 baseline | 0.5537 | 0.7136 |
| | **NPU INT8** | **0.5315** | **0.6952** |
| | 양자화 손실 | **−4.0%** | −2.6% |
| **yolo11l** | fp32 baseline | 0.5690 | 0.7240 |
| | **NPU INT8** | **0.5539** | **0.7112** |
| | 양자화 손실 | **−2.7%** | −1.8% |

→ INT8로 fp32 대비 **~96~97% mAP 유지**(INT8 검출기 전형 손실 1~5%). 4모델 모두 동일 파이프라인·calib(val2017 200장).
**정확도 vs 속도**: 11l이 mAP 최고(0.554)·양자화 손실 최소지만 채널당 5.1ms, 11n은 0.42·1.9ms. 실시간·고채널이면 11s/11m가 균형점.

## 6. 재현

```bash
# 컴파일 (env: yolo_c = qbcompiler + torch 2.7.1 + ultralytics, tutorial/yolo_npu/README.md 2절)
python -m yolo_npu.compile --model yolo11m --schemes single,multi,global4,global8 \
    --calib <coco/val2017> --calib-num 200 --out ./yolo_out
# 이 리포트의 배치/카드 벤치 (env: pe_npu_host) — 2·3절 전부 재생성
python reports/scripts/bench_yolo_all_sizes.py --section all
#   → reports/assets/yolo_all_sizes.json (A: 코어모드×배치, B: 카드수×배치)
# 추론/데모
python tutorial/yolo_npu/demo_yolo11_npu.py --mxq yolo_out/yolo11m_single.mxq --image bus.jpg
# mAP
python reports/scripts/eval_yolo_map.py <mxq> <onnx> <val2017> <instances_val2017.json> 300
```

*실측 ARIES2, qbcompiler 1.1.2, qbruntime. 2·3절 배치지연은 NPU 7장(aries0~6) 서버, 순수 NPU 추론(전/후처리 제외), median of 3, 카드당 8스레드 동기 infer. latency는 calib 무관(random 입력), 정확도는 val2017 200장 calib. mxq: `scratch_yolo/final/`(HF `PIA-SPACE-LAB/MXQ_NPU/yolo/<model>/<scheme>/`).*
